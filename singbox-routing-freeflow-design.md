# 星盘路由与免流架构设计

> 本文用于先固定架构，再改代码。当前结论：**应用是否进 TUN**、**进 TUN 后走代理/免流/混合策略**、**目的地规则分流** 是三层不同决策，不能混在一个开关里。

---

## 1. 目标

- 保持 sing-box 原生 TUN 单进程主线。
- 增加两个用户可配置出口：`proxy` 与 `free-flow`。
- 保留 `direct`，但它是系统自动内置出口，不是用户需要配置的第三个节点。
- 应用层同时支持：
  - 是否进入 TUN：继续使用黑/白名单模式。
  - 进入 TUN 后的策略：自动/混合、仅代理、仅免流。
- 免流与代理可分别开关，且二者都开启时可配置优先级。
- 没有可用节点或配置不完整时拒绝启动，避免断网、空转、无限重启。
- WebUI 配置尽量按需加载，避免应用列表和设置页卡顿。

---

## 2. 术语

| 名称 | 含义 |
|------|------|
| `direct` | 自动直连出口。由模块保证存在，用于私网、国内绕过、兜底直连等场景。 |
| `proxy` | 普通代理出口。由用户导入节点后生成 selector/urltest/节点集合。 |
| `free-flow` | 免流出口。由用户自行配置免流节点或特殊出站参数。 |
| TUN 名单 | 决定某个应用流量是否进入 sing-box TUN。对应 `packages.include` / `packages.exclude`。 |
| 应用策略 | 应用已经进入 TUN 后，决定它参与代理规则、免流规则，还是二者混合。 |
| 目的地规则 | 进入某个策略后，再按域名/IP/规则集决定走对应出口或直连。 |

---

## 3. 三层决策模型

### 3.1 第一层：是否进入 TUN

这一层只回答一个问题：**这个应用的流量是否交给模块处理？**

- 黑名单模式：默认进 TUN，`packages.exclude` 中的应用不进 TUN。
- 白名单模式：默认不进 TUN，只有 `packages.include` 中的应用进 TUN。
- 不进 TUN 的应用：模块不处理它的数据和 DNS。
- 进 TUN 的应用：继续进入第二层“应用策略”。

白名单模式需要在 UI 提示：只有白名单里的应用会进入 TUN，旁边的应用策略下拉框只对这些应用生效。

### 3.2 第二层：应用策略

这一层只对已经进入 TUN 的应用生效。

建议新增两个文件：

- `packages.proxy`：强制只走代理策略的应用。
- `packages.free-flow`：强制只走免流策略的应用。

应用策略：

| 策略 | 文件表示 | 含义 |
|------|----------|------|
| 自动/混合 | 不在 `packages.proxy` / `packages.free-flow` | 同时允许代理规则和免流规则命中，顺序由全局优先级决定。 |
| 代理 | 在 `packages.proxy` | 只应用代理规则，不应用免流规则。 |
| 免流 | 在 `packages.free-flow` | 只应用免流规则，不应用代理规则。 |

如果同一个包名同时出现在 `packages.proxy` 和 `packages.free-flow`，应在保存或启动前拒绝，不能猜测优先级。

### 3.3 第三层：目的地规则

这一层决定已经选中“代理策略”或“免流策略”的流量最终去哪。

代理规则初始建议：

- `off`：不启用代理规则。
- `global`：全部走 `proxy`。
- `bypass-cn`：私网/CN 直连，其余走 `proxy`。

免流规则初始建议：

- `off`：不启用免流规则。
- `global`：全部走 `free-flow`。

后续可以扩展免流域名/IP/规则集列表，但第一版先把架构打正，不在 shell 里手搓复杂规则解析。

## 4. 推荐配置项

`settings.env` 建议新增或调整为：

```env
SBMAGIC_BOOT_START=false
SBMAGIC_PROXY_RULE_MODE=bypass-cn
SBMAGIC_FREE_FLOW_RULE_MODE=off
SBMAGIC_MIXED_RULE_PRIORITY=proxy
```

说明：

- `SBMAGIC_BOOT_START=false`：开机自启动默认关闭，避免刷入模块后自动接管网络。
- `SBMAGIC_PROXY_RULE_MODE=bypass-cn`：保持当前“国内/私网直连，其余代理”的默认行为。
- `SBMAGIC_FREE_FLOW_RULE_MODE=off`：免流默认关闭，用户明确配置后再启用。
- `SBMAGIC_MIXED_RULE_PRIORITY=proxy`：混合模式下默认代理优先。免流关闭时不会产生额外影响。

需要删除当前半成品里的错误模型：

- 不要继续使用 `SBMAGIC_DOMESTIC_OUTBOUND`。
- 不要把 `SBMAGIC_FREE_FLOW` 做成单个布尔开关。
- 不要把“国内直连/绕过大陆”和“免流出口”混成同一个概念。

---

## 5. 路由顺序

路由应遵循“先应用策略，后目的地规则”的顺序。

建议生成顺序：

1. DNS 流量：`hijack-dns`。
2. sniff。
3. IPv6 禁用时的拒绝规则。
4. 私网/LAN：全局 `ip_is_private -> direct`，排在所有代理/免流策略前。
5. `packages.proxy`：只展开代理规则块。
6. `packages.free-flow`：只展开免流规则块。
7. 自动/混合应用：按 `SBMAGIC_MIXED_RULE_PRIORITY` 展开代理/免流规则块。
8. 最终兜底：`direct`。

注意：

- 强制代理/强制免流的应用规则必须排在混合规则前面。
- 私网/LAN 恒直连不属于代理或免流策略，不应被 `global` 代理、`global` 免流或混合优先级覆盖。
- sing-box route 是首个命中生效，所以前面的应用级规则会自然截断后续混合规则。
- 如果某个规则模式为 `off`，不要生成对应规则块。
- 如果混合模式中某一路关闭，就只生成另一路。
- 如果代理和免流都关闭，最终应落到 `direct`；但如果用户点击启动且没有任何有效出口，启动前应提示配置不完整，而不是静默运行一个“看起来启用了但实际全直连”的模块。

---

## 6. 启动前校验

配置保存和服务启动要分开处理。

保存配置时：

- 可以允许用户先保存不完整的节点草稿。
- 仍要做语法和引用校验，例如 JSON 格式、tag 重复、selector 引用不存在。

启动或重载时：

- 必须做运行态就绪校验。
- 如果代理规则可能命中，则 `proxy` 必须存在，且不能只是指向 `direct` 的空 selector。
- 如果免流规则可能命中，则 `free-flow` 必须存在，且不能缺少真实出站。
- 如果某个应用被强制设为代理，但代理规则关闭或节点缺失，应拒绝启动。
- 如果某个应用被强制设为免流，但免流规则关闭或节点缺失，应拒绝启动。
- 如果包名同时存在于 `packages.proxy` 和 `packages.free-flow`，拒绝启动。

建议把复杂 JSON 检查放到 `magic-fetch` 这类 Go 工具中做，shell 只负责调用结果，避免用 grep 猜 JSON。

## 7. WebUI 设计

### 7.1 应用页

每一行应用应有两个独立控件：

- 是否进入 TUN：沿用当前黑/白名单点击逻辑。
- 应用策略下拉框：`自动` / `代理` / `免流`。

行为：

- 下拉框默认 `自动`。
- 选择 `代理`：写入 `packages.proxy`，并从 `packages.free-flow` 移除。
- 选择 `免流`：写入 `packages.free-flow`，并从 `packages.proxy` 移除。
- 选择 `自动`：从两个文件都移除。
- 点击下拉框不应触发行点击选择。

白名单模式 UI 提示：

> 当前为白名单模式，只有已勾选应用会进入 TUN；未进入 TUN 的应用策略不会生效。

### 7.2 设置页

新增设置：

- 开机自启动：`SBMAGIC_BOOT_START`，默认 `false`。
- 代理规则模式：`off` / `global` / `bypass-cn`。
- 免流规则模式：`off` / `global`。
- 混合优先级：`proxy` / `free-flow`。

UI 文案要避免把“启用免流”和“选择免流出口”混成一个东西：

- 开关或规则模式控制是否生成免流规则。
- 出站配置控制 `free-flow` 这个出口是否可用。
- 应用策略控制某个应用是否只走免流策略。

### 7.3 性能

应用列表不要在首页初始化时加载。

建议：

- 进入“应用”页时才加载 per-app 配置和应用列表。
- 进入“设置”页时才加载 settings。
- 应用列表渲染分批进行，搜索和筛选加 debounce。
- 不要在页面启动时同步应用列表。
- 不要用短间隔轮询刷设置状态，用户动作触发即可。

## 8. 开机自启动

`magicctl boot` 应先读取 `SBMAGIC_BOOT_START`。

- `false`：只记录日志并退出，不启动 sing-box。
- `true`：继续走正常启动流程。

`SBMAGIC_ENABLED` 仍表示模块服务是否启用，用于手动启动/停止和 watchdog 退出判断。

两者关系：

| 配置 | 含义 |
|------|------|
| `SBMAGIC_ENABLED=false` | 服务关闭，手动启动也不应继续运行。 |
| `SBMAGIC_ENABLED=true` + `SBMAGIC_BOOT_START=false` | 允许用户手动启动，但开机不自动启动。 |
| `SBMAGIC_ENABLED=true` + `SBMAGIC_BOOT_START=true` | 开机自动启动。 |

## 9. 断流风险检查

需要重点避免的断流点：

- `proxy` selector 只有 `direct`，用户以为代理已启用，实际没有可用代理。
- `free-flow` 被规则或应用策略引用，但没有对应 outbound。
- 启动时配置不完整，watchdog 反复拉起又失败，造成无限重启。
- IPv6 关闭但没有接管/拒绝 IPv6，导致 IPv6 绕过代理泄漏或偶发卡顿。
- DNS 远程解析走代理，但代理节点域名又依赖远程解析，形成鸡生蛋。
- reload 失败时先杀旧进程，导致原本可用连接被坏配置打断。
- UI 保存时把规则模式、应用策略、出站 tag 混在一起，生成不可预期路由。

处理原则：

- 启动前拒绝坏配置。
- reload 先渲染和 check，通过后再重启。
- 看门狗只处理进程死亡，不承担配置热重载。
- 配置变更通过用户事件触发，不用频繁轮询。

## 10. 实施顺序

1. 清理错误模型：删除 `SBMAGIC_DOMESTIC_OUTBOUND` 和布尔式 `SBMAGIC_FREE_FLOW`。
2. 增加默认文件：`packages.proxy`、`packages.free-flow`。
3. 增加设置项：开机自启、代理规则模式、免流规则模式、混合优先级。
4. 在 `magicctl` 中补齐配置文件映射、校验、status 输出。
5. 重写 route 生成，按本文“三层决策模型”输出规则。
6. 增加运行态就绪校验，只在 start/reload/boot 前阻止启动，不阻止用户保存草稿。
7. 更新 WebUI 应用页：TUN 勾选和策略下拉分离。
8. 更新 WebUI 设置页：新增规则和开机自启设置。
9. 做 UI 按需加载和应用列表分批渲染。
10. 运行校验、打包、再处理 release。
